# Do You Even Need a Database?

> Автор: Jay | DB Pro Blog | 15 апреля 2026

## Тезис

База данных — это просто файлы. SQLite — один файл на диске. PostgreSQL — директория файлов с процессом перед ними. Вопрос не в том, использовать ли файлы — вы всегда их используете. Вопрос: свою файловую систему или базу данных?

Для множества приложений, особенно на ранних стадиях, ответ может быть: **свою**.

## Эксперимент

Построили один HTTP-сервер на Go, Bun и Rust. Два варианта хранения:
1. Читать файл при каждом запросе
2. Загрузить всё в память при старте

Инструмент: `wrk` — 4 потока, 50 соединений, 10 секунд, случайные GET-запросы.

## Три подхода к хранению

### Approach 1: Читать файл при каждом запросе

JSONL-файлы: `users.jsonl`, `products.jsonl`, `orders.jsonl` — newline-delimited JSON, одна запись на строку, append при записи.

- Запись: O(1) — просто добавить строку
- Чтение: O(n) — линейный скан всего файла
- Подходит для: очень маленьких объёмов, редких чтений

### Approach 2: Загрузка всего в память

При старте читаем все файлы в `sync.Map` (Go), `Map` (Bun), `HashMap` (Rust). RwLock/sync.RWMutex для параллельных читателей.

- Запись: append в файл + обновление in-memory map
- Чтение: O(1) в памяти
- Недостаток: RAM масштабируется с данными

### Approach 3: Бинарный поиск на диске

Файл данных сортируется по ID. Рядом — индексный файл фиксированной ширины: 58 байт на запись (`<36-UUID>:<20-значный-offset>\n`).

Binary search по индексу = O(log n) seek операций (≈20 для 1M записей), затем одно точное чтение из файла данных.

> Merge pattern — это и есть LSM-tree (Log-Structured Merge-tree).

## Результаты бенчмарков

| Размер данных | Подход | RPS | Latency p99 |
|---|---|---|---|
| 10k records | File-per-request | низкие | высокая |
| 10k records | In-memory | высокие | низкая |
| 10k records | Binary search | средние | средняя |
| 100k records | File-per-request | очень низкие | очень высокая |
| 100k records | In-memory | высокие | низкая |
| 100k records | Binary search | средние | средняя |
| 1M records | In-memory | деградирует | растёт |
| 1M records | Binary search | стабильные | стабильные |

**Вывод:** In-memory не масштабируется. Binary search — золотая середина.

## Когда база данных НЕ нужна

- < 100k записей
- Простые модели данных (один тип сущности)
- Не нужны сложные запросы (JOIN, агрегации)
- Один сервис, нет распределённого доступа
- early-stage прототипы

## Когда база данных НУЖНА

- Сложные связи между сущностями
- Много читателей и писателей одновременно (ACID)
- Нужны индексы, полнотекстовый поиск
- Масштаб: миллионы записей
- Резервное копирование, репликация
- Несколько сервисов обращаются к одним данным

## Ключевая мысль

> "Большинство приложений меньше, чем люди думают. Начните с файлов. Переходите на БД когда реально нужно."
